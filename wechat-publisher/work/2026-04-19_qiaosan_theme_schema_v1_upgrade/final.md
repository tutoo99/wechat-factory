---
title: 我把公众号排版从一个 theme.yaml 拆成了 5 个文件，多套风格这次算是接住了
---

前几天我在折腾公众号发布链路时，又被排版这件小事狠狠干了一次。

表面上看，我当时只是想多接几套风格：技术号想要更密一点的笔记感，观点文想要更柔一点的杂志感，后面顺手还能按文章类型自动推荐。

结果一动手才发现，真正卡住我的不是 CSS，不是 HTML，也不是“再加两个主题”这么简单。  
真正的问题是：**我之前那套排版系统，从数据结构上就不适合继续长大。**

以前我只有一个 `theme.yaml`。

能跑的时候，它确实很省事。颜色、字号、标题、引用、代码块，全塞进去，一个文件改完就结束。

但只要你想把它从“一个能用的主题配置”升级成“一个能长期演进的排版系统”，问题就全出来了：

- 设计 token 和业务元信息混在一起
- 渲染器能读，但推荐器很难读
- 想加一种 block 变体，只能继续往一个大 yaml 里堆字段
- 你知道它长什么样，但你不知道它**适合什么文章**

我后来越看越觉得，这个问题跟前几天我把发布系统改成 `channel` 模型很像。

**看起来是配置不够，实际上是建模错了。**

## 一开始我只是想多加几套风格

最初的诉求其实非常朴素：

1. 现在的技术号排版有点单调
2. 以后想接入多套风格
3. 还能根据文章类型自动推荐

如果只是应付眼前需求，更直接的办法当然是继续在 `theme.yaml` 上打补丁。

比如再加几组颜色：

```yaml
colors:
  primary: "#D4553B"
  secondary: "#E07C4F"
  accent: "#F4A261"
```

再加几个开关：

```yaml
section_title_style: underline
quote_style: left-bar
code_style: dark-terminal
table_style: compact-grid
```

然后命令行里继续传：

```bash
python3 md_to_styled_html.py final.md -o article.html -t tech-clean
```

短期看，这么干完全没毛病。

但我这次想做的，不只是“再多两个 theme”。  
我真正想做的是下面这几件事一起成立：

- 旧命令继续能跑
- 新主题可以按统一结构扩展
- 主题本身能描述适用赛道、人设、文章类型
- 流水线选完框架后，能顺手给出排版建议
- 正文定稿后，可以直接按推荐主题生成 `article.html`

只要目标一变，你就会发现：  
原来那个单文件配置，不是“不够优雅”，而是**职责已经塞爆了**。

## 真正的问题，不是 theme 少，而是 theme 不会“说话”

我后来回头看旧系统，更大的问题不是样式弱，而是主题信息根本没有分层。

以前那个 `theme.yaml`，本质上只服务一件事：**让渲染器画出页面。**

这意味着它最擅长回答的是：

- 标题用什么颜色
- 正文字号多大
- 代码块背景是什么

但它没法很好回答另外一类问题：

- 这个主题更适合技术号还是情感号？
- 它更适合收藏型长文，还是观点型短文？
- 遇到代码块多的文章，它是不是应该优先被推荐？
- 遇到关系文、故事文，它是不是应该避开？

也就是说，旧主题只能“被调用”，不能“被理解”。

这在单主题时代不是问题。  
一旦你想做多主题、多风格、自动推荐，它立刻就会变成问题。

因为未来系统里真正重要的，不只是“怎么渲染”，而是“为什么推荐这个主题，而不是另一个”。

如果主题自己都说不清自己适合谁，推荐器就只能靠硬编码瞎猜。

这就很危险了。

今天你还能靠记忆记住：

- `tech-clean` 适合技术号
- `emotion-warm` 适合情感号

但后面多到 8 套、12 套、20 套之后呢？

只靠命名约定和人脑记忆，迟早会失控。

## 所以我把一个 theme.yaml 拆成了 5 层

这次我最后落下来的结构，是一个 theme pack v1。

目录长这样：

```text
themes/<theme-id>/
├── manifest.yaml
├── tokens.yaml
├── blocks.yaml
├── heuristics.yaml
└── preview.md
```

这 5 个文件看起来像是“拆复杂了”，但它解决的是一个很核心的问题：

**把原来混在一起的 4 种职责，终于拆开了。**

### `manifest.yaml`：回答“我是谁，适合谁”

这个文件不负责样式细节，它只负责主题的身份信息。

比如：

- 主题 ID
- 显示名
- 适配赛道
- 适配 persona
- 适配文章类型
- 适配传播目标
- 微信安全级别

这层一拆出来，推荐器就终于有东西可读了。

它不需要再去猜 `dense-note` 这个名字像不像技术风，只要看 `lane_fit`、`persona_fit`、`article_fit` 就行。

### `tokens.yaml`：回答“页面长什么样”

这一层放设计 token，比如：

- 颜色
- 字号
- 行高
- 间距
- 圆角
- 边框
- 阴影

它的价值不是“看起来更专业”，而是未来你要做统一调色、统一字号策略时，不用去翻渲染器逻辑。

### `blocks.yaml`：回答“不同区块怎么长”

我这次比较想保留下来的，其实就是这一层。

因为未来多套排版风格，真正拉开差距的，往往不是全局主色，而是 block-level 的处理方式：

- 标题是下划线还是色块
- 引用是左边栏还是卡片
- 代码块是浅底还是终端风
- 表格是紧凑网格还是柔和面板

如果没有这一层，你最后做出来的“多主题”，很容易只是换皮。

看起来变了，读起来没变。

### `heuristics.yaml`：回答“什么时候推荐我，什么时候别找我”

这层就是给推荐器准备的。

里面会放一些偏好和避让规则，比如：

- 偏好包含代码的文章
- 偏好长文
- 避免高情绪密度内容

这样推荐就不再只是“按 channel 粗暴映射”，而是开始有一点内容感知。

### `preview.md`：回答“这套主题到底有没有跑偏”

这个文件我一开始觉得是附属品，后来发现它很关键。

因为一旦你有多套主题，就需要一个最小回归样例，去看下面这些东西有没有坏掉：

- 标题
- 列表
- 引用
- 代码块
- 表格
- 分割线

否则你每改一次渲染器，都只能靠肉眼随机抽查，迟早翻车。

## 这次很关键的一步，不是拆文件，而是先做兼容层

我很清楚一件事：这种基础设施改造，最怕的不是写不出来，而是**一升级就把旧链路干挂了**。

所以我这次没有直接废掉旧 `theme.yaml`，而是先做了一个兼容 loader。

它的逻辑很简单：

1. 优先读新结构的 `manifest.yaml`
2. 如果没有，就回退读旧 `theme.yaml`
3. 再把旧配置归一成 schema v1 的内存对象
4. 最后让渲染器统一只吃这一种结构

这一步特别像我前面做 `channel` 模型时的一个判断：

**升级路径要先保证系统还能活，再追求系统更漂亮。**

如果没有这层兼容，你所有旧命令、旧主题、旧脚本都得一起改。

工程上这不是升级，这是爆破。

有了兼容层之后，事情就顺很多：

- 旧入口 `md_to_styled_html.py` 继续能用
- 新入口 `render_wechat_article.py` 可以放心往前走
- 老主题可以慢慢迁
- 新主题直接按 v1 新建

我这次甚至顺手加了两个新主题：

- `dense-note`：更适合技术号收藏型内容
- `magazine-soft`：更适合观点文、关系文这种柔一点的表达

这时候多主题系统才算真正开始有点样子。

## 我觉得这波升级值回票价的地方，是推荐链路接上了

如果这次只是把主题结构拆漂亮了，我不会这么兴奋。

真正让我觉得这波升级值钱的，是它已经不再停留在“主题仓库重构”，而是跟写作流水线连上了。

现在我的链路大概是这样：

1. 先根据选题推荐框架
2. 选中框架后生成 brief
3. 顺手产出一份 `*.theme.yaml`
4. 正文定稿后，直接按推荐主题渲染 `article.html`

也就是说，排版不再是最后人工拍脑袋选一个主题。  
它开始变成流水线里一个有依据的决策节点。

比如技术号这类“踩坑复盘型”文章，系统现在就更容易推荐：

- `dense-note`
- `tech-clean`

如果后面是情感号、关系文、故事型内容，它再偏向：

- `emotion-warm`
- `magazine-soft`

这个变化看起来不大，但其实很重要。

因为它把“排版风格”从一个纯手工审美选择，往前推成了一个**可沉淀、可复用、可自动化的工程决策**。

这一步一旦成立，后面你再做：

- 风格市场
- 原文样式对标学习
- 主题 AB 测试
- 按内容类型自动切换 block 变体

才是顺着路往前走，而不是继续打补丁。

## 这次我给自己沉淀了一个判断标准

如果你也在改一个“刚开始很好用，后来越改越拧巴”的系统，我建议你记住这条判断标准：

**当一份配置同时想服务渲染、推荐、兼容、扩展这 4 件事时，它大概率已经不该继续是一个文件了。**

换句话说，不要只看“现在还能不能用”，要看它是不是还适合继续长大。

我这次这套 theme schema v1，本质上不是为了“文件更规范”，而是为了把未来几条路一次性打通：

- 多套排版风格
- 自动主题推荐
- 旧主题平滑迁移
- block 级风格扩展
- 排版预览与回归测试

这几个能力如果分开做，看起来都是小优化。  
但放在一起，你就会发现，它们背后其实是在要同一件东西：

**一个能表达自己、也能被系统理解的主题模型。**

## 所以这次我修的不是样式，而是一条演进路径

很多时候我们做升级，容易只盯着眼前那个痛点。

比如这次表面上的痛点，明明只是：

- 排版风格有点单调
- 想接多套样式

但如果我真的只按这个表面需求去修，大概率就是再多加几个 yaml 字段，然后半年后重来一遍。

这次我比较庆幸的一点，是我最后没停在“再加两个主题”这一步，而是顺手把整个排版系统的演进路径也修了一遍。

所以现在回头看，我比较满意的其实不是某个主题好不好看，而是现在可以比较踏实地说一句：

**这条路后面能继续走了。**

旧链路还能跑。  
新主题可以继续长。  
推荐器有数据可读。  
流水线可以自动收口。  
以后真要接 10 套、20 套排版风格，也不至于第一步就把自己写死。

如果你最近也在重构一套内部工具，尤其是那种“刚开始就自己用，后来越接越多”的系统，我建议你多问自己一句：

你现在修的，到底是一个功能点，还是一条未来的路？

这个问题想明白，很多重构决策会一下子清楚很多。
