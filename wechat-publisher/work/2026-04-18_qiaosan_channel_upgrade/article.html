1️⃣ 先说结论：这次改的不是名字，是模型

前两天我在排查公众号发布链路的时候，表面上看是在改几个目录名。

account1 改成 mp__qiaosan，test_account 改成 mp__wannianqingxingshuo，好像只是把名字起清楚一点。

但真往下挖，问题根本不在名字，而在模型。

之前这套系统里，profile、内容类型、文章归档账号 这几个概念是缠在一起的。写文章的时候想的是“这篇是技术号还是情感号”，发布的时候却要去记“该用哪个 Firefox 目录”，排查问题的时候又要看“这篇文章归档在 tech 还是 emotion”。

三个层次混成一层，系统短期能跑，长期一定乱。

这次我下手重构，核心就一句话：把发布身份、内容约束、业务入口拆开。

2️⃣ 以前为什么容易乱

以前最容易出问题的地方，有两个。

第一个是正文和发布素材混在一起。

文章正文写完后，标题备选、摘要、导语、封面文案也一起塞进 final.md，结果转 HTML 的时候全被带进正文里。你以为自己在发一篇技术文，实际发出去的是“技术文 + 发布素材大全”。

第二个是“你以为你在选账号，实际上你在选浏览器目录”。

比如你脑子里想的是：“这篇发乔三技术号。”

但系统里真正让你选的却是 account1。

这中间至少隔了三层翻译：

  • account1 是哪个 profile
  • 这个 profile 登录的是哪个公众号
  • 这个公众号对应的是哪套内容约束

只要你隔一阵子不用，或者后面号一多，就一定会出现“我知道要发哪个号，但我不记得该用哪个 profile”的情况。

说白了，这不是记忆力问题，是系统抽象有问题。

3️⃣ 这次我怎么拆

这次我把整个链路拆成了三层。

第一层是 profile。

它只负责一件事：保存公众号后台的登录态。

所以 profile 目录名现在只表达“这个目录登录的是哪个公众号”，比如：

  • mp__qiaosan
  • mp__wannianqingxingshuo

第二层是 persona。

它负责的是内容约束，也就是这篇文章该怎么写，用什么口吻，举什么例子，哪些句式不能碰。

比如：

  • persona-tech.yaml
  • persona-auntie.yaml

以后如果我新增法律号、八卦号、穿搭号，也不是去改 profile，而是新增：

  • persona-law.yaml
  • persona-gossip.yaml
  • persona-fashion.yaml

第三层是 channel。

这是这次最关键的新层。

channel 不保存登录态，也不直接写文章，它做的是“业务映射”：

  • 这篇文章属于哪个入口
  • 这个入口该加载哪套 persona
  • 最后发布时该走哪个 profile
  • 归档时该落到哪个 account 目录

所以现在我用一个 channels.yaml 来统一管理。

技术号这条链路大概就是：

  • channel = tech
  • persona = tech
  • profile = mp__qiaosan
  • archive_account = tech

情感号就是：

  • channel = emotion
  • persona = auntie
  • profile = mp__wannianqingxingshuo
  • archive_account = emotion

这样一来，写文章的时候我选的是 channel，不是 profile。

我只需要回答一个业务问题：这篇文章发哪个号？

后面的 profile、persona、归档目录，系统自己顺着映射往下走。

4️⃣ 这一改，几个老毛病一起没了

这次重构完之后，几个之前总反复出现的问题，基本都顺手被收掉了。

第一个，正文和发布素材分开了。

现在 final.md 只放正文，发布素材单独存。就算有人手滑把素材附在正文后面，转换脚本也会在生成 article.html 时自动剥离。

第二个，首页多刷一次的问题没了。

之前脚本打开公众号后台后，在登录检测里又 page.get() 了一次首页，相当于自己给自己补了一刀刷新。现在这个多余动作已经去掉，链路顺滑很多。

第三个，重复发布有了保护。

之前 token 失效重新登录之后，如果上层流程又触发了一次 publish，系统并不会拦，结果同一篇稿子会在短时间内再发一次。

现在我加了短时间内相同产物的重复发布检测。10 分钟内同一篇 HTML、同一张封面、同一份 Markdown，再来一遍就直接拦住。除非我明确传 --force，否则不会傻乎乎再发一轮。

这类保护看着不高级，但特别有用。

很多自动化系统真正难的，不是“让它第一次跑通”，而是“让它在异常场景下别乱跑”。

5️⃣ 我对这套系统现在更放心的原因

我现在对这条链路更放心，不是因为它功能更多了，而是因为它终于更像一个能维护的系统了。

以前你新增一个账号，脑子里要同时记四件事：

  • 用哪个 profile
  • 对应哪个内容类型
  • 文章归档去哪
  • 这套人设文件叫什么

现在新增一个号，流程清楚很多：

  1. 新建或复用一个 profile
  2. 新建一份 persona（如果是新赛道）
  3. 在 channels.yaml 里加一条映射

这就结束了。

账号换赛道也不用动登录态。比如一个号今天写技术，后面改写法律，我只需要把 channel 里的 persona 从 tech 改成 law，profile 目录完全不用碰。

这个设计的好处，不是优雅，是省事。

省的是后面半年、一年里每次维护时的脑力税。

6️⃣ 这次升级最大的收获，不是修了 bug

很多时候我们做系统升级，第一反应都是修 bug、补功能、加命令。

但这次我最大的收获其实不是这些。

而是我再次确认了一件事：一套系统能不能长期跑，不取决于你加了多少功能，而取决于你有没有把概念边界切清楚。

profile 就该只管登录态。

persona 就该只管内容约束。

channel 就该只管业务入口和映射。

边界一清楚，命名自然顺了，流程自然顺了，后面加号、换号、转型、排查问题，都会轻很多。

所以这次表面上是把公众号系统升级了一轮，实际上更像是给自己补了一次系统设计课。

以前是“先跑起来再说”。

现在是“跑起来之后，把结构理顺，别让未来的自己继续替今天的偷懒买单”。
