前两天那篇写 AI Agent 提速复盘的文章，24 小时阅读过万，后台私信也炸了。

但大家问得最多的，不是"这个 prompt 怎么写"，也不是"现在该用哪个模型"，而是一个更扎心的问题：

**为什么同样都在学 AI、都在折腾工具，有的人三个月了还没打出一个像样结果，有的人一个周末就把自动化链路跑通了？**

我这两年从手工测试一路踩坑到现在用 Agent 搭 pipeline，说实话结论挺不浪漫的：

**不是你能力不够，是你根本没进入"全力投入"的状态。**

而且这事跟加不加班没关系。我见过天天熬夜但产出为零的，也见过每天只花两小时但一条链路直接跑通的。区别就在一个地方——

## 你是"尽力了"，还是"拼尽全力"

这两句话看着像一回事，其实差了一个维度：**这件事搞砸了，最后谁来兜底。**

你只是参与者，你的动作会不自觉地变虚：

- prompt 改两下觉得"差不多行了"
- 报错了先绕过去，回头再说
- 方案能跑通就收工，不会多想一步"万一重试怎么办"
- 结果不稳定，安慰自己"AI 本来就这样"

但你是最后那个兜底的人，脑子会立刻清醒。你会开始问：这条链路最脆的环节在哪？重复执行会不会炸？回滚点放哪？凌晨三点自动跑出错了谁来看？

**这才叫投入。**

还有一种更隐蔽的——有些人开会特别积极，群里回消息特别快，谁焦虑他先给一波安慰，看起来像团队里最上心的人。但说句不好听的，超额提供情绪价值不一定是负责，也可能是在用"存在感"换"结果感"。让你觉得自己参与很多，实际上一个硬结果都没落地。

## 全力投入 ≠ 更辛苦，而是更负责

我一开始也觉得全力投入就是死磕。后来发现不是。

做 AI 相关的工程，最怕的恰恰是**半懂不懂就开干**。因为模型很会给你一种错觉——"它已经写了这么多代码，看起来挺像那么回事，应该差不多能用了。"

但你只要真的背过锅、接过线上事故，就知道差得远。

真正拉开差距的，其实是这几个很朴素的东西：

- 你喂给 Agent 的上下文够不够（不是丢个项目目录就完事）
- 你有没有先把约束找清楚（到底卡在准确率、稳定性、还是重复执行）
- 你是不是盯着它一步步跑，不让它热心发挥
- 你有没有补验证闭环（失败了怎么回滚、重复执行是否一致）

说白了就是一件事：**你在对结果负责，还是在完成流程。**

我自己的做法现在基本就是这 5 步，不复杂：

```text
吃透上下文
  ↓
找到真正约束
  ↓
写清目标和非目标
  ↓
让 Agent 执行，但你是总调度
  ↓
每一步强制验证
```

这套东西一点都不玄学，本质就是把"我在认真做"升级成"我对结果负责"。

## 具体怎么落地，我说几个我踩过的坑

### 坑 1：公众号发布链路，真正难的不是"发出去"，是"别重复发"

我原来的发布链路能发，但不够稳。一遇到重试、标题微调、封面改动，去重逻辑就开始飘。有时松有时紧，一出问题我只能人工翻日志，想补发也没有明确的 override 流程。

这类问题最烦——它不是完全不能用，而是半死不活地给你找麻烦。

我当时的做法是：先把最近 7 次发布日志、两次异常重试记录、旧的去重规则一起喂给 Agent，让它先研究，不准直接改。第一轮它给的方案太大，想顺手把状态机也重构了。我直接砍掉，只保留这次必须做的：指纹字段重算、去重窗口控制、告警日志、手动 override、验证脚本。

前后盯了大概 2 小时 20 分钟。其中改代码的时间不长，更多花在盯边界和补验证上。

上线之后，一次发布从人工盯 15-20 分钟变成 3-5 分钟看一眼日志就够了。最关键的收获不是"又修了个 bug"，而是这条链路从"勉强能用"提到了"我敢持续用"——这就是全力投入和随便搞搞的区别。前者留下资产，后者留下情绪。

### 坑 2：银行项目接口回归，难的不是生成用例，是生成后还能维护

一个银行项目，接口多、字段杂，每次回归前大家要花大量时间看文档、补参数、改断言、更新 token。就算用 AI 生了第一版，也经常是第一轮能跑、第二轮改了接口就全散了。

我当时没让它"帮我生成所有测试"——那样八成翻车。我给的是更窄的指令：基于现有接口说明和历史请求样例，先生成 20 个核心接口的 pytest 回归骨架，公共鉴权单独抽离，环境变量配置化，断言至少覆盖状态码和关键业务字段。

第一轮 Agent 又想全量铺开，我又把它拉回来，只准先出 `conftest.py`、`api_client.py` 和 smoke 用例。

这次最值钱的动作不是它写了多少代码，而是我盯着它把公共层先抽对了。前后 3 小时。按以前纯手工，至少两天。

但这里有个关键——如果我只是"试试看 AI 能不能生成用例"，心态就完全不一样，大概率就是一堆跑不通的废代码。当你把自己放到"这套回归后续我要接着维护"的位置上，你盯着的方式就不一样了。

### 坑 3：微信侧数据整理，半吊子投入只会让你永远停在脚本玩具阶段

我早期做微信自动化就是这样：手动导数据、手动改格式、临时写个脚本、下一次再来一遍。脚本一堆，看起来挺忙，没留下什么真正能复用的东西。

后来我换了思路。不再问"怎么最快搞一个脚本"，而是自己先画清最小流程（格式检测→清洗→异常落表→标准输出→人工复查），然后让 Agent 按这个做一版最小链路。明确告诉它不许顺手加 dashboard、不许做可视化、不许抽成通用框架。

这次省下来的不是编码时间，而是我不再每周都重新想一遍这件事了。原来一周要花 3-4 次碎片时间处理，现在集中一次 25-40 分钟跑完。同类任务后续复用时，直接复制目录结构就能开工。

很多效率型项目最大的敌人不是难，是永远停在半自动。你没全力把它收成资产，它就一直吞你的注意力。

## 几个我觉得有用的具体做法

不是方法论，就是我自己踩坑踩出来的习惯。

### 写 non_goals.md

以前我觉得"目标都写了，非目标还写啥"。后来被 Agent 和自己教育了很多次——没写非目标，等于默认什么都可能被顺手做掉。

```md
# 本次不做
- 不重构无关模块
- 不顺手升级依赖
- 不新增 UI
- 不处理历史脏数据
- 不为了优雅牺牲交付时间
```

就这几行，能砍掉一半以上"看起来努力其实没用"的动作。

### 给关键动作加保护

公众号发布、定时消息、Webhook、批量写库这类活，没保护早晚会翻车。我现在用的最小配置就是：

```yaml
publish_guard:
  dedupe_window_minutes: 180
  fingerprint_fields:
    - channel
    - title_normalized
    - content_hash
  on_duplicate: block
  on_block:
    log_level: warn
    notify: true
  manual_override:
    enabled: true
    reason_required: true
```

平时像没用，真出一次重复发布你就知道它值多少了。

### 让 Agent 每一步都带验证

我现在开跑前会追加一句要求：每完成一个阶段，同时输出"改了什么、为什么这么改、怎么验证、失败了怎么回退"。

别小看这个。很多"AI 写的代码不能用"的坑，不是因为模型不行，是因为你没有把验证绑定在执行后面。

## 算笔账

很多人不敢全力投入，觉得成本高。但我自己算下来，最贵的从来不是工具费，而是反复试错和烂尾。

拿上面几个场景粗算一下：

| 场景 | 以前 | 现在 |
| --- | --- | --- |
| 公众号发布 | 反复试错 + 人工盯 | 从"勉强能用"到"稳定可复用" |
| 接口回归 | 手工搭骨架，改一次疼一次 | 3 小时出第一版，后续扩展 5-8 分钟/接口 |
| 微信数据 | 每周零散手动处理 | 一次 25-40 分钟跑完 |

所以如果问我工具选择，态度很简单：一个工具能让我更快吃透上下文、稳定拆任务、减少返工，我愿意花。只是多一个聊天窗口，没兴趣。

如果你现在只想先上一个主力工具试试，我个人用的是 `Claude Max`，入口在这：[Claude Max 官方入口](https://www.anthropic.com/max)。不是推荐你买，先看看它能不能解决你的问题，能用再考虑。套餐的话：轻度用先上基础档验证自己会不会真用，重度天天要喂文档拆任务的直接上高配，已经有 Cursor 的别急着换全套，先补一个长上下文研究位就行。

先把一条链路打穿，再谈升级。

---

说到底，**很多事没有结果，不是你不努力，是你的投入强度根本没到能打穿结果的那条线。** 做技术、做自动化、做 AI 流程，半吊子投入最贵——它会让你一直觉得自己快成了，实际上一直没真正成过。
