前两天我在用 buildmate-engine 跑一个阶段一拆帖入库流程，用户已经把原贴给全了，规则也写得很清楚，结果还是卡了快 1 个小时。

最尴尬的不是慢。

是中间用户已经催了好几次“继续”“还没结束？”，而我这边很多回合其实没干活，只是在汇报、解释、再分析。

这次把我打醒的点是：很多 AI Agent 看起来像是“工具慢”“脚本慢”，其实真正拖时间的，是它在长上下文里反复推理、反复吃自己、然后停在中间等人催。

后面我没去调模型参数，先把流程改了。目标很简单：把原来一条容易跑到 1 小时级的链路，强行压回 3 轮主路径。

这篇就复盘下，我是怎么定位这个坑，又是怎么改的。

01这次坑，不是没能力做完，而是一直在路上打转

先说下现场。

当时要做的事情其实不复杂：读原贴，按固定六步拆解，生成结构化 case，然后跑注册、校验、索引。

真正有价值的动作就这么几步：

  1. 读原文
  2. 生成结构化稿
  3. 跑 pipeline
  4. 交付结果

但实际执行时，链路被拉得很长：

  • 先去读大段 skill 文档
  • 再去读 schema
  • 路径不确定，再 search 一轮
  • 看目录结构
  • 看脚本源码
  • 用户催一下，先汇报进度
  • 再分析一下为什么慢
  • 再继续执行

最后你会发现，真正的执行动作没几个，绝大部分时间都花在“准备执行”和“解释为什么还没执行”上了。

这个状态，做过 agent 的人应该都不陌生。

你明明感觉系统已经知道该干嘛了，但它还是忍不住先去多读一点、多看一点、多想一层。然后上下文越吃越大，后面每一轮又更慢。

02真正的大头，不是 tool 慢，是 LLM 在反复重吃上下文

我后面重新看了这次流程，发现一个很扎心的事实：

shell 执行本身其实很快，真正贵的是 LLM 回合数。

每多一轮 tool call，模型就要把前面那一坨上下文再吃一遍。

而且越到后面越惨，因为上下文不是固定的，它会越来越胖：

  • 前面读过的 SKILL.md
  • 读过的 workflow
  • 读过的 schema
  • 脚本 help
  • 错误输出
  • 自己上一轮说过的话

这些东西全会重新进入下一轮推理。

所以很多人看到 agent 慢，第一反应是：

  • 是不是脚本太慢
  • 是不是命令跑太久
  • 是不是环境卡

但这次我基本可以确定，主因不是这些。

主因是：成功路径不够窄，导致模型在本来已经可以直接干活的时候，还在不停探索。

你可以把它理解成一个人去厨房炒个蛋炒饭。

锅、蛋、米饭都在面前了，他不先开火，反而先去翻一遍《中华料理大全》，再确认一下冰箱有几层，再去研究燃气灶说明书，最后还回头给你汇报一句：“我正在处理中。”

这就不是做饭慢，这是空转。

03空转的根因，其实是“协议漂移”

这次我总结下来，根因主要有三个。

第一，成功路径太宽

阶段一明明只需要一条很短的成功链路：

  1. 读 fast path
  2. 读输入原文
  3. 生成结构化稿
  4. 跑 pipeline

结果之前的设计里，SKILL.md、workflow.md、schema.md、脚本源码都很容易被顺手读进去。

一旦成功路径允许“顺手再看点别的”，它就一定会越跑越宽。

第二，进度汇报和执行动作被拆开了

这个特别坑。

很多时候模型会先来一句“我正在处理”“我现在在第几步”，但这一回合结束后，它并没有接着执行下一动作。

于是用户看到的是：有汇报，没推进。

从交互体验上看，这比直接沉默还难受，因为你会误以为它已经在干活了。

第三，失败路径和成功路径混在了一起

正常情况应该是：

  • 成功时，走极简路径
  • 真失败了，才升级诊断

但很多 agent 的文档和流程不是这样设计的。

它会把“排查失败时需要的信息”也默认塞进“成功时的准备动作”里。

最后就变成：还没失败，先按失败预案把整套资料都看一遍。

这也是典型空转。

04我后面没先换模型，先把流程砍成了 3 轮

这次改造我做得很克制，核心思路就一句话：

先把成功路径做窄，再谈模型能力。

我主要做了 5 个动作。

1. 把 skill 改成导航页，不再当百科全书

以前一个 skill 文件里什么都想放，阶段一、阶段二、反馈、校验规则，全塞一起。

后面我把它收成更像“路由器”：

  • 你现在走阶段一，就先看 stage1-fast-path
  • 你走阶段二，再去看阶段二文档
  • 你做反馈，再去看反馈文档

这样模型一进场，不会先背一整本说明书。

2. 单独做一份 stage1-fast-path

这个文件只干一件事：告诉模型，阶段一成功路径到底该怎么最短地跑完。

里面我直接写死：

  • 默认只读 fast path 和用户输入
  • 不默认读 workflow.md
  • 不默认读 schema.md 全文
  • 不默认读脚本源码
  • 成功路径固定为 1/4 -> 2/4 -> 3/4 -> 4/4

这一步很关键。

因为很多 agent 不是不会做，而是没有被明确约束“不要多做”。

3. 直接内联最小 case 模板

之前为了写一个 case，模型经常会回去读完整 schema。

后面我把成功路径真正需要的最小结构，直接塞进 fast-path：

  • frontmatter 需要哪些字段
  • 正文最少要哪些 section
  • 是否推测 只能写 是/否

这样成功路径就不必每次都回头吃完整 schema 了。

4. 把成功路径和失败路径彻底拆开

我现在的规则是：

  • 没失败，不准扫目录
  • 没失败，不准读大段源码
  • 没失败，不准做中途复盘
  • 只有脚本真的报错了，才允许进入升级排查

而且升级顺序也写死：

  1. 先看错误文本
  2. 白名单格式错，先自动修
  3. 还不行，再定点读一个相关文件

不是一上来就“把能看的都看一遍”。

5. 强制“汇报必须绑定下一动作”

这是我觉得最该加的一条。

后面我把协议写成了硬约束：

有合法输入后，进度汇报必须和下一执行动作出现在同一回合。

换句话说，不能只说：

“我现在在阶段一第 2 步，预计还要 3 分钟。”

然后停住。

你要么别汇报，要汇报完就继续干下一步。

这条规则非常土，但非常有用。

05这套方法，其实可以直接拿去改你自己的 Agent

如果你现在也在做 agent workflow，我建议你先别急着调 prompt 的语气，也别急着换更大的模型。

先检查 5 件事：

1. 成功路径是不是一句话能说清

如果一个任务的成功路径，你不能用 3 到 5 个步骤讲清楚，那大概率已经太宽了。

2. 成功时会不会默认读失败预案

如果会，那就是典型上下文浪费。

3. 进度汇报是不是会单独占一个回合

如果会，用户体感一定差，而且很容易出现“看起来在动，其实没动”。

4. 最小必需结构有没有被内联

能写在 fast path 里的最小模板，就别逼模型每次回头翻完整 schema。

5. 错误升级顺序是不是固定的

如果失败后允许模型自由探索，它很快就会从“定点排查”变成“全仓库旅游”。

06最后一句

这次最有价值的结果，不是我把一条链路优化了。

而是我更确定了一件事：

很多 AI Agent 的慢，不是能力不够，而是工作流把它带偏了。

模型当然重要，但在很多真实场景里，先决定上限的，反而是你有没有把“该走的路”和“不该走的路”切干净。

路径不干净，模型越强，空转得越高级。

路径收窄了，普通模型也能跑得很利索。

这事我自己也是踩坑后才真的信。

你现在手上的 Agent，有没有那种“明明知道该干嘛，却总想先研究一下环境”的毛病？评论区说说你最烦的一次空转场景。

下一篇我准备继续拆：怎么给 Agent 设计“成功路径 / 失败路径”双轨协议，尽量让它少读文档、少扫目录、少等人催。
