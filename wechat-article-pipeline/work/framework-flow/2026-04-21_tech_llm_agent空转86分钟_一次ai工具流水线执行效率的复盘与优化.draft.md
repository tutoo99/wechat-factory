那天我本来只是想偷个懒。

我把一个内容生产任务丢给 Agent，要求很简单：读 persona，套 framework，整理素材，输出一篇技术号初稿。按我之前的经验，这种活儿 10 到 15 分钟差不多就该收工了。我就去接了个水，顺手回了两条消息，回来一看，终端还在刷。

再一看时间，已经过去了 86 分钟。

更扎心的是，它不是卡死，也不是报错，而是在一种很“勤奋”的状态里持续空转：一会儿重读约束，一会儿改提纲，一会儿自我复盘，一会儿又说“为了提高质量，需要进一步验证结构完整性”。看起来很努力，实际一行能交付的正文都没落下来。

这次踩坑之后我才真正意识到，很多人说 Agent 不稳定、LLM 不靠谱，其实只说对了一半。很多低效，不是模型能力问题，是你的流水线设计本身就容易让它空转。

## 这 86 分钟，具体耗在哪了

我后面把执行日志从头捋了一遍，发现这 86 分钟基本浪费在 4 件事上。

第一段浪费，是任务边界太糊。

我当时给的指令是大而全的，类似这种：

- 先理解选题
- 再吸收 persona
- 再按 framework 组织
- 再补足细节
- 再自查风格
- 最后输出正文

人看着没问题，Agent 看着就容易出事。因为这里面同时混了“理解”“规划”“生成”“审查”4种动作，但没有明确说每一步什么时候结束。

结果就是它一直在“准备开始写”，但永远没真正开始写。

第二段浪费，是上下文喂太多了。

persona 很长，framework 也有要求，外加任务说明、排版偏好、渠道信息、目标动作，一股脑全塞进去。理论上信息更全，实际上它每一轮都在重新消化一遍这些约束。

你可以把这理解成一个刚入职的同事，每次干活前都重新翻员工手册、项目背景、写作规范、历史案例。不是他不努力，是你没给他一个足够短、足够明确的任务卡。

第三段浪费，是没有 stop condition。

这是最致命的。

Agent 一旦没有“做到什么程度就必须停”的规则，它就会默认继续优化。尤其是碰到写作这类开放任务，它特别容易滑进一个死循环：

- 提纲还能再顺一点
- 例子还能再补一点
- 语气还能再像真人一点
- 结尾还能再收束一点

每一句都对，但加起来就是灾难。因为它不会像人一样在第 12 分钟突然醒悟：“差不多得了，先交一版。”

第四段浪费，是让同一个 Agent 既当作者又当审稿人。

这事我以前在测试里就踩过坑。你让一个系统同时负责“生成”和“否定自己”，它很容易进入反复推翻的状态。

这次也是一样。正文刚起了个头，它就开始审自己；审完觉得不够像技术号，又退回去重写；重写完觉得案例不够完整，又回头扩写前文。整个链路没有前进，只有原地打转。

## 表面上看是模型慢，实际上是流水线没长脑子

很多人复盘 Agent 效率时，第一反应是：

- 模型不够强
- 上下文窗口不够大
- 提示词不够细
- 工具不够多

这些当然有影响，但不是这次问题的核心。

这次真正的问题，是我把一个需要“分段执行”的流程，交给了一个“默认持续思考”的系统，还没给它任何刹车。

说白了，我把 Agent 当万能实习生用了。

但 Agent 不是人。它没有“差不多就行”的直觉，也没有“先交付再优化”的职业本能。你不给它状态边界，它就会一直算；你不给它退出条件，它就会一直改；你不给它角色切分，它就会在一个线程里自己跟自己打架。

所以这次真正有价值的，不是最后把时间从 86 分钟压下来，而是我把这件事想明白了：

**Agent 流水线的核心，不是让它更聪明，而是让它更容易停下来。**

## 我后来怎么改的

我没有去换模型，也没有去堆更复杂的 prompt，先动的是流水线结构。

第一刀，拆阶段，不让一个 Agent 干完所有事。

我把原来那条大而全的链路，改成了 4 个阶段：

## 1. Intake：只做任务收敛

这个阶段只回答 3 个问题：

- 这次要交付什么
- 哪些约束必须遵守
- 哪些信息缺失但可以先假设

它不能写正文，不能提前优化语言，不能扩展任务。

产物也很固定，就是一张任务卡，类似这样：

```markdown
目标：输出公众号正文初稿
必须遵守：persona口吻、framework结构、只输出Markdown正文
已知素材：选题、channel、goals、framework
可做假设：案例细节可基于真实工作流合理补全
禁止动作：提前润色、重复解释任务
```

这一步看起来土，但真香。因为一旦任务卡被压成这个体积，后面的 Agent 就不用每轮都重读一大坨背景。

## 2. Planner：只出结构，不写正文

这个阶段只负责回答：

- 开头怎么起
- 案例怎么展开
- 抽象方法落在哪
- 结尾怎么钩子收束

输出也限制成固定骨架，最多一屏，不允许自由发挥成长篇分析。

我后来发现，很多空转不是出在“不会写”，而是出在“边想边写边推翻”。先把结构钉住，后面生成就顺很多。

## 3. Writer：只按骨架写，不回头开会

真正写正文的阶段，我只给它三样东西：

- 任务卡
- 结构骨架
- 少量必要约束

我刻意不把完整历史对话再塞回去。因为写作阶段最怕的，不是信息少，而是噪音多。

这个阶段还有一个硬规则：

**写到能交付就停，不做自我二次评审。**

这条规则救了我很多时间。

## 4. Reviewer：只检查，不重写

最后单独做一个 review 阶段，但权限很小，只能做 3 类检查：

- 有没有违背 persona
- 有没有跑偏 framework
- 有没有明显啰嗦、重复、结构断裂

如果发现问题，它只能指出“哪一段有问题，为什么”，不能整篇接管重写。

这样做的好处是，writer 不会被 reviewer 拖回去无限返工；reviewer 也不会一上头，把一篇能交付的稿子改成另一个项目。

## 真正起作用的，不是拆阶段，而是加了预算和止损线

拆阶段只是第一步。很多人拆完照样慢，因为他们忘了第二件更重要的事：给每个阶段上预算。

我后来给每个节点都加了 3 个限制。

第一，时间预算。

比如：

- Intake 最多 2 分钟
- Planner 最多 3 分钟
- Writer 最多 8 分钟
- Reviewer 最多 3 分钟

超时就停，不准“再想一下”。

第二，重试预算。

每个阶段最多重试 1 次。第一次失败，可以针对性补一条指令；第二次还不行，直接 fallback，不让它在一个错误方向上越跑越远。

第三，输出预算。

每个阶段只能产出指定格式，不能借着“解释一下思路”偷长度。尤其是 Planner 阶段，一旦它开始写长篇分析，基本就说明又要空转了。

这 3 个预算加上去之后，我第二天拿同类型任务重跑，整条链路落到 18 分钟左右。不是每次都这么稳，但至少从“看天吃饭”变成了“可控交付”。

更关键的是，人开始知道该在哪一步介入了。

以前 86 分钟的那次，我其实中间好几次想打断，但不知道它到底是在接近完成，还是在离题万里。现在有了阶段和预算，任何一个节点超线，我都能立刻判断：该停了，该改了，不用再赌它“下一轮可能就好了”。

## 这次复盘后，我固定下来一条方法

如果你也在搭 Agent 流水线，我建议先记住一句话：

**不要按“功能”设计 Agent，要按“状态转换”设计 Agent。**

这句话有点像测试思路。

很多人设计流水线时，脑子里想的是：

- 它要会搜集资料
- 它要会写
- 它要会审
- 它要会改

听起来很完整，但这其实是“功能清单”。

真正能跑起来的流水线，应该想的是：

- 当前处于什么状态
- 这个状态允许做什么
- 输出什么才算完成
- 完成后跳到哪个下一个状态
- 失败时是重试、降级，还是转人工

一旦你开始按状态机思路设计，很多空转会自然消失。因为 Agent 不再拥有无限解释权，它只是被放进一个个窄通道里往前走。

这也是为什么我现在越来越少写那种“一个 prompt 干全套”的大提示词了。演示时挺唬人，真落地到生产链路里，经常就是高配版空转机。

## 我现在判断一条 Agent pipeline 能不能上线，只看 5 件事

这 5 条是我这次填坑之后固定下来的检查项，基本适用于写作、测试、脚本生成、报告总结这些常见流程。

## 1. 这一步的输入是不是最小必要集

如果一个节点必须吃完整历史对话才能工作，基本已经埋雷了。能压成任务卡，就不要喂原始上下文。

## 2. 这一步的输出是不是固定格式

没有固定输出格式，Agent 就容易开始“展示思考过程”，然后越写越多，越多越偏。

## 3. 这一步有没有明确完成条件

比如“输出一版可读初稿”是完成，“继续优化表达”不是完成。后者是无底洞。

## 4. 失败后有没有默认处理动作

是重试一次、降级输出、还是转人工？如果没定义，系统就会默认无限自救。

## 5. 这个节点如果连续两轮没前进，能不能立刻停

这是最实用的一条。别迷信它“再来一次就好了”。同一个节点连续两轮没有明显新增产出，十有八九已经进入空转区了。

你可以把这 5 条当成一个很粗暴的判断标准：

**凡是不能回答清楚这 5 个问题的 Agent 流水线，先别谈提效。**

## 以后怎么复用，这事不只适合写作

我后面把这套方法拿去改了另外几条链路，效果都不错。

比如接口测试这类任务，我现在会把流程拆成：

- 解析需求
- 生成测试点
- 生成脚本
- 执行校验
- 汇总报告

这里面“解析需求”和“汇总报告”可以放给 LLM，“执行校验”和“格式转换”尽量脚本化。能 deterministic 的部分就别交给 Agent，别让它在本该走规则的地方发挥创作热情。

再比如做日报、周报、问题复盘，我也不再让它从一堆聊天记录直接“写一篇完整报告”，而是先抽事件，再归类，再判断重点，最后才生成正文。链路一旦拆开，稳定性会明显高很多。

所以回到最开始那个 86 分钟的事故。

如果只看结果，这就是一次低效执行；如果往上抽一层，它其实给了我一个很实在的提醒：

**Agent 真正适合做的，不是“包打一切”，而是“在被约束好的阶段里高速前进”。**

约束越清楚，交付越快。
角色越单一，空转越少。
止损线越明确，系统越像工具，而不是像个会一直说“我再想想”的同事。

现在我自己判断一条链路值不值得继续投入，标准也变得很简单：

**不是看它能不能跑，而是看它会不会自己停。**

不会停的 Agent，再聪明，也只是更贵的空转。
