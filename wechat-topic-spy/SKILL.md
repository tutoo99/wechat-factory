# 每日选题侦察与爆文筛选

## Overview

每天从采集到的爆文数据中，筛选出最适合你账号定位的选题。你只需要提供原始数据，AI自动完成筛选和推荐。

## 数据输入方式

用户提供采集好的 CSV 或 Excel 表格文件。**统一一种格式，不需要截图或纯文本。**

### 表格字段规范

| 字段 | 必填 | 说明 |
|------|------|------|
| 文章URL | 是 | 微信公众号文章链接 |
| 对标账号 | 是 | 发这篇文章的公众号名称 |
| 阅读量 | 是 | 文章阅读数 |
| 点赞数 | 是 | 点赞+在看数 |
| 转发数 | 是 | 转发/分享数 |
| 收藏数 | 否 | 有则填，没有留空 |
| 评论数 | 否 | 有则填，没有留空 |

**数据来源**：
- **系统推荐流采集**：微信"看一看"、搜一搜、订阅号推荐流中看到的好文章，复制链接后统一整理到表格
- **对标账号采集**：定期把对标账号的最新文章整理到表格

### 输入方式

- 把 CSV/Excel 文件路径发给 Hermes，或直接粘贴表格内容
- Hermes 读取后先校验字段完整度，缺失"必填"字段的行标记为"数据不全"，单独列出提醒补充，不参与评分
- 通过文章URL自动抓取标题和正文内容用于选题分析

## 爆文判断标准

以下标准用于判断一篇文章是否值得作为选题参考：

### 阅读量标准

| 等级 | 阅读量 | 说明 |
|------|--------|------|
| 小爆 | 1万+ | 基本门槛 |
| 爆文 | 10万+ | 真正的爆文标准 |
| 大爆 | 50万+ | 头部爆文 |

### 互动数据标准

| 指标 | 计算公式 | 理想值 |
|------|---------|--------|
| 点赞率 | 点赞数 / 阅读量 | > 1% |
| 转发率 | 转发数 / 阅读量 | > 3% |
| 收藏率 | 收藏数 / 阅读量 | > 1%（无数据跳过） |
| 评论率 | 评论数 / 阅读量 | > 0.1%（无数据跳过） |

### 综合爆文标准

同时满足以下条件，判定为爆文：
- 阅读量 10万+
- 点赞数 > 1000
- 转发数 > 500
- 点赞率 + 转发率 > 5%

### 行业差异调整

不同行业的爆文标准不同：
- **情感类、娱乐类**：互动率天然偏高，标准可以适当提高
- **科技类、财经类**：互动率相对较低，标准适当降低

## 选题筛选流程

### 第1步：加载账号人设

读取对应账号的 `persona-<标识>.yaml`，确认：
- 叙述者身份
- 读者画像
- 内容方向
- 语气风格

### 第2步：分析爆文数据

对用户提供的每篇爆文，提取：
- 选题方向（一句话概括）
- 情绪按钮（读者看了会有什么感受）
- 结构类型（并列式/递进式/对比式/故事式）
- 是否适合本账号人设

### 第2.5步：赛道一致性检查（前置过滤）

在匹配度评分之前，先检查选题是否与账号当前赛道一致。

**核心原则：赛道一致性检查的前提是"这个赛道被爆文验证过"。没有爆文验证的赛道，保持一致没有意义——你只是在错误方向上保持一致。**

#### 决策流程

```
这个赛道有没有爆过（进过流量池）？
├── 有 → 锁定赛道，不允许跨（规则A）
└── 没有 → 进入"赛道验证中"状态
    ├── 已发 < 10篇 → 继续坚持，暂不换（样本不够）
    ├── 已发 ≥ 10篇 → 警告"赛道可能没选对，建议换方向"（规则B）
    └── 换赛道之后 → 从0开始重新验证，至少坚持10篇再判断（规则C）
```

#### 规则A：赛道有爆文保护 → 不允许跨

当赛道已被爆文验证（有文章阅读量进入流量池），说明算法已经打准了标签。此时跨赛道 = 扔掉好不容易建立的算法标签。

- 系统对账号标签还在建立期，每发一篇都在告诉算法"这个号是做什么的"
- 算法根据你发的内容给账号打标签，决定把文章推给谁
- 换赛道 = 算法还在用旧标签推人，新内容推给了错误的读者，互动数据暴跌，算法判定"内容不好"然后停止推荐
- 赛道本身没有大小之分，每个赛道都有流量池，关键是让算法把标签打准，然后持续喂同类内容
- 即使一个选题很火，如果它属于另一个赛道，在标签建立期也不该写

#### 规则B：赛道无爆文 + 已发 ≥ 10篇 → 建议换赛道

持续10篇以上都是几十、十几阅读，说明：
- 赛道可能没选对，或者
- 内容质量不行，或者
- 选题方向跟读者需求不匹配

此时应该主动建议用户考虑换赛道，而不是继续"保持一致"。

#### 规则C：换赛道后至少坚持10篇

换赛道不是"每篇都试一个新方向"，而是有意识地切换到一个新赛道后持续验证。至少10篇才能判断新赛道行不行，否则陷入"每篇都换赛道"的恶性循环。

**什么是跨赛道：**

| 当前赛道 | 跨赛道举例 |
|----------|-----------|
| tech（技术号） | 突然写职场情绪/个人成长 |
| emotion/female_growth（女性成长） | 突然写育儿/情感挽回/婆媳关系 |
| emotion/intimate_relationship（亲密关系） | 突然写原生家庭/职场 |
| emotion/social_insight（社会洞察） | 突然写育儿/情感鸡汤 |

**什么是同赛道内的角度拓展（允许）：**

| 当前赛道 | 允许的角度拓展 |
|----------|---------------|
| tech | 从自动化工具拓展到AI提效、学习方法、认知升级（都属于"技术人的成长"） |
| female_growth | 从身材接纳拓展到职场偏见、自我价值、不被定义的人生（都属于"女性成长"） |
| social_insight | 从财富真相拓展到历史冷知识、社会现象解读、阶层分析（都属于"你不知道的信息"） |

**判断标准：** 选题的核心读者是否与账号目标读者是同一群人。如果读者群变了，就是跨赛道。

**标签建立期后（20篇+）：** 可以偶尔跨赛道，但建议不超过每5篇中1篇，且跨赛道选题必须是强爆款潜力。

**反面案例：「阳子在成长」的赛道混乱**

阳子第一篇《跨越三次羞耻》爆了（5k+），算法给她打了"女性成长"标签。之后她连发两篇跨赛道内容：
- 第2篇写"孩子考98分治愈了我"（中年妈妈感慨，偏育儿情绪）
- 第3篇写"培养牛逼女孩这三点"（纯粹的育儿/教育文）

结果：第2篇开始掉池，第3篇及之后连续个位数阅读。

注意两点：
1. 阳子的写作能力没问题（第一篇有真实故事、有细节、有转折），问题纯粹是赛道乱了
2. 她大概率没意识到自己在做"女性成长号"，只是碰巧第一篇爆了就跟着感觉写——这也是大多数人会犯的错：没有"这个号要持续输出什么"的清晰认知

**教训：赛道一致性不能靠感觉判断，必须靠规则强制约束。创作者自己觉得"差不多算同赛道"没用，算法怎么标签你才是关键。**

### 第3步：匹配度评分

对每篇爆文打分，评分维度：

| 维度 | 权重 | 说明 |
|------|------|------|
| 读者匹配度 | 30% | 选题是否踩中目标读者痛点 |
| 人设适配度 | 25% | 叙述者能不能自然地写这个话题 |
| 传播潜力 | 25% | 是否有转发/收藏/讨论的价值 |
| 差异化空间 | 20% | 能不能写出跟原文不一样的角度 |

### 第4步：输出选题候选

按综合评分排序，输出3-5个最推荐的选题。

## 输出格式

```markdown
# 每日选题候选 - YYYY-MM-DD

## 数据来源
- 用户采集表格：X篇（数据完整X篇，数据不全X篇）

## 爆文分析
| # | 标题 | 对标账号 | 阅读 | 点赞 | 转发 | 评分 | 适合度 |
|---|------|----------|------|------|------|------|--------|
| 1 | xxx | xxx | 10万 | 2000 | 800 | 85 | ⭐⭐⭐⭐⭐ |

## 推荐选题

### 选题1：[一句话描述]
- 来源：某爆文（标题）
- 情绪按钮：xxx
- 切入角度：xxx（跟原爆文不同）
- 推荐指数：⭐⭐⭐⭐⭐
- 推荐理由：xxx

### 选题2：...

## 今日建议
最推荐选题X，理由：xxx
```

## 第5步：拆解结果自动写入飞书

每篇爆文分析完成后，将拆解的关键信息自动写入飞书多维表格。

### 配置

读取 `references/feishu-config.yaml` 获取 base_token、table_id 和字段映射。

### 写入时机

在第2步（分析爆文数据）中，每篇文章分析完毕后立即写入。不要等所有文章都分析完再批量写入——边分析边写入，避免中途失败丢失数据。

### 写入命令

`record-upsert` 没有按业务键自动查重的功能（传 `--record-id` 走更新，不传走创建）。所以必须先搜索再条件写入：

```bash
# Step 1: 搜索是否已存在（按文章链接关键词搜索）
EXISTING=$(lark-cli base +record-search \
  --base-token <base_token> \
  --table-id <table_id> \
  --json '{"keyword":"<URL中的唯一片段>","search_fields":["文章链接"]}' \
  -q '.data.items[0].record_id')

# Step 2: 存在则更新，不存在则创建
if [ -n "$EXISTING" ]; then
  lark-cli base +record-upsert \
    --base-token <base_token> --table-id <table_id> \
    --record-id "$EXISTING" --json '<JSON对象>'
else
  lark-cli base +record-upsert \
    --base-token <base_token> --table-id <table_id> \
    --json '<JSON对象>'
fi
```

用 URL 中唯一片段（如微信文章的 `s/xxx` 部分）作为搜索关键词，避免长 URL 匹配问题。

### 写入字段

```json
{
  "文章标题": "原标题",
  "公众号": "公众号名称",
  "文章链接": "https://mp.weixin.qq.com/...",
  "拆解日期": "2026-04-29",
  "赛道": "emotion",
  "使用框架": "框架ID 框架名称",
  "标题公式": "反常识钩子+痛点描述",
  "可复用标题模板": "为什么{}的人，都{}了",
  "Hook模式": "反常识开头+自黑",
  "情绪触发": "焦虑+共鸣",
  "结构类型": "三层递进",
  "适合选题": ["情绪共鸣", "认知升级"],
  "适合目标号": "emotion-female_growth",
  "爆文级别": "10万+",
  "参考价值": "高-可直接模仿",
  "金句摘录": "摘录的金句",
  "拆解备注": "核心发现和可借鉴点"
}
```

### 注意事项

- `适合选题` 是多选字段，值必须是数组
- `赛道`、`结构类型`、`爆文级别`、`参考价值`、`适合目标号` 的值必须是字段中已存在的选项，不在选项内的值会被忽略。如果需要新增选项，先用 `+field-update` 添加选项再写入
- `拆解日期` 格式为 `yyyy-MM-dd`
- 写入失败时打印错误信息但不中断流程，继续分析下一篇文章
- 每篇文章写入后打印确认信息：`✅ 已写入飞书：<文章标题>`

## 选题库管理

每日选题自动追加到选题库：`~/.hermes/output/wechat/<账号>/topics/topic-database.md`

格式：
```markdown
# 选题库

## 2026-04-14
- [选题描述] 来源：xxx | 评分：85 | 赛道：female_growth | 状态：已发 | 阅读：5200 | 日期：04-14
- [选题描述] 来源：xxx | 评分：80 | 赛道：female_growth | 状态：已发 | 阅读：30 | 日期：04-15
- [选题描述] 来源：xxx | 评分：75 | 赛道：female_growth | 状态：待写
```

**字段说明：**
- 赛道：该选题所属赛道（tech / emotion/female_growth / emotion/intimate_relationship 等）
- 状态：待写 / 已发 / 跳过
- 阅读：发布后的阅读量，发布后由用户回填
- 日期：发布日期

**发布后数据回填（极轻量）：**

用户发布文章后，只需一句话回填阅读量：
- "这篇5000阅读" → 自动更新对应选题的 阅读 字段
- "这篇30阅读" → 同上
- "这篇没流量" → 自动记录 阅读：0

系统根据选题库自动判断赛道状态（不需要用户提供额外数据）：
- 当前赛道发了多少篇 → 数"已发"状态的同一赛道记录
- 有没有爆过 → 检查同一赛道是否有阅读量进入流量池的记录
- 自动匹配规则A（有爆文→锁定）/ 规则B（无爆文+≥10篇→建议换）

当用户说"看看选题库有什么没写的"时，从选题库中筛选状态为"待写"的选题推荐。
